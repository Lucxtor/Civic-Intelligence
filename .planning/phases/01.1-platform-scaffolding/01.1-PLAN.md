# Plan: Phase 1.1 - Platform Scaffolding

## Phase Overview
The goal of this phase is to scaffold the reactive functional shell of the Civic Intelligence Platform. This includes all RESTful routes and high-fidelity UI components for proposal management, discovery, and community feedback. 

Identity is handled via a simplified Web3 wallet demographics model.

## Component Breakdown

### 1. Infrastructure & Global State
- **Goal**: Enable session-persistent, backend-less data flow with RESTful routing.
- **Tasks**:
  - `01.1-01`: Configure **Wagmi** and **RainbowKit** in root layout.
  - `01.1-02`: Expansion of `useMockStore.ts` with `persist` middleware.
  - `01.1-03`: Global **Breadcrumb & Active Proposal** indicator in header.
  - `01.1-03b`: **Auth Wall Logic**: Block access to `/admin` and `/proposals/[id]/vote` if wallet is not connected, but allow public reading.

### 2. The Admin Proposal Forge (`/admin`)
- **Goal**: Build a high-fidelity form supporting the refined registry schema.
- **Tasks**:
  - `01.1-04`: Scaffold Admin layout as a **Single Scrolling Form** with a sticky preview sidebar.
  - `01.1-05`: Implement **Registry Form**: Title, Categories, ELI5/Technical Summaries, Full Markdown.
  - `01.1-06`: Implement **Impact Matrix Editor**: Dynamic rows for Beneficiaries and Negative Impacts; financial object builder.
  - `01.1-07`: Build the **Metric Forge**: Question builder to define community-facing facets.

### 3. Public Discovery & Reading (`/proposals`)
- **Goal**: Deliver a premium reading experience with RESTful integrity.
- **Tasks**:
  - `01.1-08`: Implement the `/proposals` Bento Box gallery. Outer cards display: **Title, Category, Impact Level, Start of ELI5**.
  - `01.1-09`: Build the **Summary Toggle** (ELI5 vs Technical) with layout-switch animations.
  - `01.1-10`: Integrated "Vote" and "Dashboard" entry points per proposal.

### 4. Interactive Feedback & Dashboard Shells
- **Goal**: Decisions mapped to demographic analysis.
- **Tasks**:
  - `01.1-11`: Build `/proposals/[id]/vote` route replacing numerical sliders with a **Likert Scale** (Strongly Disagree -> Strongly Agree). Mandatory `Approval` toggle and optional `Facet` Likerts.
  - `01.1-12`: Build `/proposals/[id]/dashboard` initialization. Use **Recharts** shell containers with "Awaiting Participation Data" states.


## Success Criteria (UAT)
1. **The Registry Loop**: Admin can define structured impact data and custom vote metrics.
2. **RESTful Consistency**: Navigating between details, voting, and dashboard preserves the "Active Proposal" context in the Breadcrumbs.
3. **Data Fidelity**: All registry fields (Impacts, Financials, Categories) render correctly in the Technical Abstract view.
4. **Theme Alignment**: Admin forge shares the same glass-panel aesthetic as the community-facing pages.

## Verification Plan
- **Manual**: Create a proposal titled "Testing Civic Infra", verify it appears on the homepage, and check if the custom metrics (e.g., "Sustainability") show up as sliders on the vote page.
- **Build**: Ensure Next.js static optimization works for the dynamic routes.
